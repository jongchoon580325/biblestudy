# Supabase 자료실 PRD (Product Requirements Document)

## 1. 목적
- 모든 자료(성경자료, 일반자료, 특별자료 등)의 업로드/조회/수정/삭제를 Supabase DB 및 Storage 기반으로 일원화한다.
- 파일 메타데이터는 Supabase Table, 실제 파일은 Supabase Storage에 저장한다.
- 클라이언트는 DB fetch 결과를 zustand 등 store에 캐싱하여 최적화한다.

---

## 2. Supabase 테이블 설계

### 2.1 테이블명: `materials`

| 컬럼명           | 타입            | 설명                  | 제약조건                |
| -------------- | -------------- | ---------------------| --------------------- |
| id             | bigint         | 자료 고유 ID (PK)      | PRIMARY KEY, auto inc |
| title          | text           | 자료 제목              | NOT NULL              |
| category       | text           | 자료 카테고리           | NOT NULL              |
| book           | text           | (성경자료) 책 이름      | NULL 허용               |
| file_url       | text           | Storage 파일 URL     | NOT NULL               |
| file_size      | integer        | 파일 크기 (byte)      | NOT NULL               |
| upload_date    | date           | 업로드 일자            | NOT NULL              |
| file_type      | text           | 파일 확장자            | NOT NULL              |
| description    | text           | 자료 설명             | NULL 허용               |
| created_at     | timestamptz    | 생성일시              | DEFAULT now()          |
| updated_at     | timestamptz    | 수정일시             | DEFAULT now()           |

#### 2.2 테이블 생성 SQL
```sql
create table public.materials (
  id bigint generated by default as identity primary key,
  title text not null,
  category text not null,
  book text,
  file_url text not null,
  file_size integer not null,
  upload_date date not null,
  file_type text not null,
  description text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- updated_at 자동 갱신 트리거
create or replace function update_updated_at()
returns trigger as $$
begin
  new.updated_at = now();
  return new;
end;
$$ language plpgsql;

create trigger trigger_update_updated_at
before update on public.materials
for each row
execute procedure update_updated_at();
```

## 3. Supabase Storage 구조
- **버킷명:** `materials`
- **폴더 구조 예시:**
  - `materials/bible/2024-06-01/파일명.pdf`
  - `materials/general/2024-06-01/파일명.xlsx`
- 파일 업로드 시 업로드 일자/카테고리별로 폴더 구분 가능
- 파일 URL 예시: `https://<project-id>.supabase.co/storage/v1/object/public/materials/bible/2024-06-01/파일명.pdf`

## 8. RLS(Row Level Security) 정책 설계

### 8.1 RLS 활성화
```sql
-- RLS 활성화
alter table public.materials enable row level security;
```

### 8.2 정책 설계
- **업로드/조회:** 인증된 사용자만 가능
- **수정/삭제:**
  - 관리자는 모든 자료 수정/삭제 가능
  - 일반 사용자는 본인이 업로드한 자료만 수정/삭제 가능
- **관리자 식별:** `auth.users`의 특정 claim(예: role=admin) 또는 별도 관리 테이블 활용
- **업로더 식별:** `materials` 테이블에 `user_id uuid` 컬럼 추가 필요 (업로더의 Supabase user id)

#### 8.2.1 user_id 컬럼 추가
```sql
alter table public.materials add column user_id uuid references auth.users(id);
```

#### 8.2.2 정책 예시

```sql
-- 인증된 사용자만 SELECT(조회) 허용
create policy "Authenticated users can select materials" on public.materials
  for select
  using (auth.uid() is not null);

-- 인증된 사용자만 INSERT(업로드) 허용, user_id는 본인만 가능
create policy "Authenticated users can insert materials" on public.materials
  for insert
  with check (auth.uid() = user_id);

-- 본인 또는 관리자만 UPDATE(수정) 허용
create policy "Owner or admin can update materials" on public.materials
  for update
  using (
    auth.uid() = user_id
    or
    (exists (select 1 from public.admins where user_id = auth.uid()))
  );

-- 본인 또는 관리자만 DELETE(삭제) 허용
create policy "Owner or admin can delete materials" on public.materials
  for delete
  using (
    auth.uid() = user_id
    or
    (exists (select 1 from public.admins where user_id = auth.uid()))
  );
```

#### 8.2.3 관리자 테이블 예시
```sql
create table public.admins (
  user_id uuid primary key references auth.users(id)
);
```

## 9. 개발 환경 RLS 설정 가이드

### 9.1 개발/테스트 환경용 정책
- 개발 중에는 실수로 인한 데이터 접근 차단/권한 문제를 빠르게 확인할 수 있도록 별도의 테스트 정책을 추가할 수 있음
- 예시: 모든 사용자가 전체 자료를 조회/수정/삭제할 수 있는 임시 정책(운영 배포 전 반드시 삭제)

```sql
-- (개발 전용) 모든 사용자가 전체 자료 SELECT 허용
create policy "Dev: allow all select" on public.materials
  for select using (true);

-- (개발 전용) 모든 사용자가 전체 자료 UPDATE 허용
create policy "Dev: allow all update" on public.materials
  for update using (true);

-- (개발 전용) 모든 사용자가 전체 자료 DELETE 허용
create policy "Dev: allow all delete" on public.materials
  for delete using (true);
```

> ⚠️ 운영 배포 전 반드시 위 정책을 삭제 또는 비활성화할 것!

### 9.2 개발자 계정/관리자 계정 관리
- 개발/테스트용 관리자 계정은 `public.admins` 테이블에 직접 추가
- 예시:
```sql
insert into public.admins (user_id) values ('개발자_유저_UUID');
```
- Supabase Auth에서 발급받은 UUID 사용

### 9.3 정책 관리 방법
- 정책은 Supabase Studio(웹 콘솔) 또는 SQL 스크립트로 관리
- 정책 변경 시 반드시 코드/문서화(버전 관리) 병행
- 운영/개발 환경 분리 권장(별도 프로젝트 또는 정책 분기)

### 9.4 테스트 체크리스트
- 인증/비인증 사용자 접근 테스트
- 일반 사용자/관리자 권한 테스트
- 본인 자료/타인 자료 수정/삭제 테스트
- 정책 적용 후 예상 동작(권한 거부 등) 확인

## 10. Supabase Storage 권한 정책 필수 가이드 (중요!)

> ⚠️ **Storage 업로드/다운로드 오류의 90%는 이 정책 미설정 때문!**
> 
> Supabase Storage 버킷(materials)이 Public이어도, 실제 파일 업로드/다운로드 권한은 반드시 "Other policies under storage.objects" 섹션에서 정책을 추가해야 정상 동작합니다.
> 
> **이 과정을 빼먹으면 프론트엔드에서 업로드/다운로드가 무조건 400/403 에러로 실패합니다.**

### [필수] Storage 업로드/다운로드 허용 정책 SQL 예시

```sql
-- 모든 사용자의 업로드 허용 (materials 버킷 한정)
create policy "Allow upload for all" on storage.objects
  for insert
  with check (bucket_id = 'materials');

-- 모든 사용자의 다운로드 허용 (materials 버킷 한정)
create policy "Allow download for all" on storage.objects
  for select
  using (bucket_id = 'materials');
```

#### [설명 및 주의사항]
- 반드시 "Other policies under storage.objects" 섹션에서 위 정책을 추가해야 함
- 버킷명 조건(bucket_id = 'materials')을 꼭 넣어야 다른 버킷에 영향 없음
- 정책 추가 후에는 브라우저 새로고침/서버 재시작 권장
- 정책 미설정 시 업로드/다운로드 모두 400/403 에러 발생 (RLS, 환경변수, DB 정책과 무관)
- 초보/주니어 개발자 실수 방지: Storage 정책은 DB 정책과 별도로 반드시 직접 추가해야 함
